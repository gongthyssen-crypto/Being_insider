# 智能体制作作业报告

## 一、项目名称

基于大模型与知识库增强的历史情境模拟智能体设计与实现

## 二、项目背景与制作动机

本次作业要求围绕“智能体制作”完成一个具备实际交互能力的 AI 应用，并提交智能体链接、讲解视频、交互记录以及项目报告。

本课程是《人工智能导论》，老师在布置作业时提到可以参考 COZE 这类智能体平台来完成，因此我在思考选题时，更关注的不是传统软件开发本身，而是如何围绕“智能体”做出一定的创意、工作量和 AI 应用深度。

我的项目选择了“历史情境模拟”这一方向，目标是构建一个能够与用户进行多轮互动、根据历史背景生成推演结果、并保持较强情境一致性的智能体。

这个项目的最初灵感来源于我在 2026 年天津 BOHACK 上看到的一个相关项目。之后我结合自己的理解和课程作业需求，对原有思路进行了进一步的魔改和扩展，把它逐步做成了一个更贴近课程要求的智能体作品。

另外，为了贴近老师提到的智能体平台方向，我也在 COZE 智能体平台上制作了一个对应版本，并为它配置了知识库。COZE 版本链接为：`https://www.coze.cn/s/7Fv9ABe9JGE/`。这样一来，我既完成了一个可以在线体验的版本，也实际体验了在智能体平台中搭建 Agent、设计工作流和组织知识库的过程。

之所以选择这个主题，主要有以下几个原因：

1. 通用聊天模型虽然可以回答历史问题，但在具体历史情境中，往往会出现背景不稳定、细节漂移、上下文不一致等问题。
2. 历史题材天然适合做智能体，因为它既需要知识支持，也需要角色理解、情境推演和多轮对话能力。
3. 我希望借这次作业完成的不只是一个简单对话框，而是一个具有明确设定、知识支撑和完整工作流的智能体。

## 三、项目目标

本项目的目标主要包括以下几个方面：

1. 构建一个可以围绕特定历史主题进行连续交互的智能体。
2. 支持用户在多个回合中输入行动、提问或决策，由系统给出带有历史背景约束的反馈。
3. 引入知识库机制，减少模型“脱离设定自由发挥”的问题，提高生成内容的一致性和相关性。
4. 在 COZE 智能体平台上完成一个可运行版本，体现对智能体平台能力的实际运用。
5. 将系统部署上线，形成一个可以直接体验和展示的作品。

## 四、系统设计与实现思路

这个项目的重点不在传统软件架构本身，而在于“智能体如何工作”。从智能体视角来看，我将整个系统理解为一个由“用户输入、场景识别、策略判断、知识检索、上下文组织、模型生成、结果反馈”组成的类工作流。

整个工作流大致可以概括为：

1. 用户输入一个行动、问题或策略。
2. 智能体先识别当前对话属于哪一个历史场景，并保证后续推演始终服务于该场景。
3. 智能体判断用户本轮的策略意图，例如更偏向稳定内政、宫廷操盘、外交周旋、军事预防、借势结盟还是权宜拖延。
4. 智能体根据当前场景自动选择对应知识库，再结合上下文检索最相关的历史资料片段。
5. 智能体围绕关键人物、主要势力、权力边界、短期收益与中短期风险进行综合裁定。
6. 大模型基于用户输入、场景设定和知识材料生成回答，当前调用的是 DeepSeek 官方平台提供的 API。
7. 智能体将结果以相对稳定的结构反馈给用户，并进入下一轮交互。

在 COZE 版本中，我也尽量沿着这个思路去组织智能体能力，把场景设定、知识库和输出方式统一到一个 Agent 工作流中，使其更符合“智能体平台应用”的表达方式。

## 五、核心功能

### 1. 历史情境模拟

用户进入指定历史场景后，可以用自然语言输入自己的行动、判断或策略，系统会结合当前背景和上下文生成反馈。

### 2. 多轮会话管理

系统会维护当前会话状态，使智能体能够延续前文，而不是每一轮都重新开始，从而提升交互连续性。

### 3. 知识库增强

我将不同历史场景对应的资料整理为知识文档，在生成回答前检索相关内容并注入模型上下文，帮助模型基于材料作答，而不是完全自由发挥。

### 4. 工作流组织

这个项目不是把用户输入直接交给模型，而是先经过场景理解、策略判断、知识检索和上下文组织，再形成一轮完整的智能体输出。这种流程化处理，是我在本次作业中比较重视的一部分。

### 5. 知识库驱动

知识库不是附属内容，而是整个智能体回答质量的关键支撑。它决定了智能体能否围绕场景稳定输出，而不是只靠模型自由发挥。同时，我也让知识库和场景绑定，即根据用户当前选择的历史场景来确定使用哪一组知识库资料。

### 6. 环境与上下文感知

这个智能体并不是只看用户当前这一句话，而是会综合当前场景、前面几轮对话推进情况以及已经发生的关键变化来组织回答。也就是说，它具备一定的环境感知和上下文感知能力，能够根据局势变化调整后续推演方向。

### 7. 自主决策与阶段总结

在设计上，我希望它不仅能“回答问题”，还能够在一定程度上自主判断当前局势应该怎样表述、哪些风险值得强调、下一步应该给出哪些方向性的建议。当一个场景已经推进数轮，或者局势已经出现比较明显的阶段性结果时，智能体还可以输出“阶段总结”，对本阶段的总体策略、局势变化和后续走向进行归纳。

### 8. 线上展示

项目已完成阿里云部署，可以通过线上地址直接访问与演示，不再局限于本地运行。

## 六、知识库设计

在项目初期，如果完全依赖大模型自由生成，虽然语言表达较为自然，但经常出现历史背景不稳定、人物行为偏离设定、上下文衔接不够紧密等问题。

为了解决这一问题，我为不同历史场景整理了相应的 Markdown 文档，作为知识库输入源。随后，我又对知识库机制进行了进一步优化，将其从简单的关键词召回升级为 embedding 检索模式。

具体流程如下：

1. 将历史文档切分为多个较小的文本片段。
2. 为每个片段生成 embedding 向量表示。
3. 在用户输入新问题或新行动时，对当前输入生成向量。
4. 根据语义相似度检索最相关的知识片段。
5. 将这些片段注入模型上下文，再生成最终回答。

这种方式相比纯关键词匹配更稳定，也更适合多轮对话中的语义检索，因此在回答质量和场景一致性上有明显提升。

## 七、智能体工作流说明

从智能体设计角度来看，我这次作业最核心的部分是工作流设计，而不仅仅是把模型接起来。整个工作流可以概括为以下几个阶段：

### 1. 场景设定

我先为智能体定义明确的历史场景，让它不是一个泛化聊天机器人，而是一个在特定历史背景下行动和回应的 Agent。

### 2. 场景识别与知识库绑定

智能体必须先确认当前对话属于哪一个历史场景，并据此确定后续要调用哪一组知识库。这样可以避免不同场景之间的资料混淆，也能让整个推演始终围绕当前时代、人物和权力结构展开。

### 3. 用户意图输入

用户每一轮输入都可能代表不同意图，例如提问、做决策、表达态度、尝试改变历史走向等。智能体需要理解这些输入的性质，而不是把所有内容都当作普通问答。

### 4. 策略判断

在理解用户输入之后，智能体还需要判断这一步主要属于哪类行动，例如稳定内政、宫廷操盘、外交周旋、对外威慑、上奏请示、军事预防、情报侦察、借势结盟、财政整顿、扩张吞并或权宜拖延。这样做的目的，是让智能体的裁定更像一种“策略分析”，而不是单纯续写一段文本。

### 5. 知识库检索与历史裁定

当用户输入进入系统后，智能体会围绕当前问题去关联历史材料。为了让回答更贴近史实和设定，我为智能体增加了知识库，使其能够结合资料进行回应。在这个过程中，智能体不仅要看材料本身，还要分析这一步会影响哪些关键人物或势力，谁会支持、谁会警惕，它是否符合该人物所处时代的权力边界，以及它会带来什么短期收益和中短期风险。

### 6. 上下文整合

除了当前输入，智能体还需要结合之前几轮对话、当前场景状态和检索到的知识片段，形成一份更完整的上下文，再交给模型处理。这一部分体现了它的上下文感知能力，也决定了多轮推演是否连贯。

### 7. 结构化输出

在综合用户输入、场景设定和知识资料之后，调用 DeepSeek 官方平台 API 生成最终输出。输出会尽量保持相对稳定的结构，例如包括“局势叙述”“结果摘要”“风险变化”“下一步建议”等部分。这样一来，智能体的回答不仅更清晰，也更像一种经过裁定后的历史推演结果。

### 8. 多轮推进与阶段总结

用户看到结果后可以继续输入下一步行动，智能体再根据新的输入重复上述流程，形成连续多轮交互。当某个场景已经推进数轮，或者局势已经形成明显阶段性结果时，智能体还可以输出“阶段总结”，对本阶段用户采取的总体策略、局势变化、收益与代价进行归纳。

### 9. 平台化实践

为了更贴近老师提到的 COZE 方向，我还在 COZE 智能体平台上做了一个版本，链接为：`https://www.coze.cn/s/7Fv9ABe9JGE/`。在这个过程中，我进一步理解了智能体平台的核心价值并不只是“提供一个聊天框”，而是帮助开发者把提示词、知识库、工作流和输出目标组织成一个可复用的 Agent。

## 九、制作过程中遇到的问题与解决方案

### 1. 模型回答容易脱离历史背景

问题表现：
在没有知识库约束时，模型虽然能生成较流畅的内容，但容易出现历史设定偏离、细节不稳定等问题。

解决方案：
增加知识库检索机制，在每轮生成前提取与当前问题最相关的历史资料片段，并注入提示词中，以增强约束能力。

### 2. 关键词检索效果有限

问题表现：
早期知识库采用关键词匹配方式，在语义相近但措辞不同的情况下召回效果较差。

解决方案：
将知识库升级为 embedding 检索模式，通过向量相似度完成语义召回，提高检索质量。

### 3. 上下文携带轮数与输出长度如何平衡

问题表现：
在多轮交互过程中，如果上下文携带的轮数太少，智能体容易忘记前面已经发生过的内容，导致角色状态和情境推进不够连贯；但如果携带轮数过多，又会让上下文变得很长，影响模型对当前问题的聚焦。同时，输出的 `max token` 如果设置得太小，回答会显得过短、信息不足；如果设置得过大，又可能出现内容冗长、重点不够集中的情况。

解决方案：
我在实际调试中不断测试不同的上下文携带轮数和输出 `max token` 参数，观察哪种组合更适合历史情境模拟这个场景。最终的思路不是一味追求“上下文越长越好”或“回答越长越好”，而是根据场景需求在连贯性、信息量和可读性之间做平衡，让智能体既能记住关键背景，又能输出相对清晰、完整且不过度发散的回答。

## 十、项目亮点

本项目的主要亮点如下：

1. 不是简单的单轮聊天，而是具备明确应用场景的多轮历史情境智能体。
2. 不仅做了一个可访问版本，也在 COZE 智能体平台上实现了对应版本，体现了对平台型 Agent 工具的实际运用。
3. 引入知识库与 embedding 检索机制，提高了知识召回效果和回答的一致性。
4. 对智能体工作流进行了较完整的设计，而不是停留在“把问题发给模型”的层面。
5. 已部署到阿里云，具有真实演示和分享价值。

## 十一、项目不足

虽然项目已经实现了预期的主要功能，但仍存在一些不足：

1. 当前知识库规模还比较有限，历史场景覆盖数量不多，内容深度也有待扩展。
2. 虽然加入了知识库增强，但生成结果仍然会受到大模型本身稳定性的影响。
3. 当前会话记忆更偏向于单次会话管理，还没有实现更长期、更个性化的记忆能力。
4. 系统对模型返回格式仍有一定依赖，虽然加入了修复机制，但不能完全消除所有异常情况。
5. 如果后续继续迭代，还可以进一步强化智能体角色感、任务拆解能力和平台化编排能力。

## 十二、反思与收获

通过这次项目，我更深刻地理解了“智能体”与“单次调用大模型”之间的区别。一个真正可用的智能体，并不是简单地把用户输入发送给模型并显示结果，而是需要同时考虑以下几个方面：

1. 上下文如何组织。
2. 知识从哪里来。
3. 多轮会话如何保持连续性。
4. 知识库怎样真正服务于智能体而不是流于形式。
5. 如何把智能体做成一个可展示、可体验、可复用的作品。

我也认识到，大模型并不是接上就一定能直接得到稳定结果。很多问题并不完全来自模型能力本身，而是与智能体设计是否合理密切相关。例如，没有知识库时模型容易跑偏；没有清晰工作流时输出就会缺少层次；没有场景约束时，智能体很容易变成普通聊天机器人。

这次作业不仅让我完成了一个可上线访问的智能体应用，也让我积累了从需求设计、功能开发、调试优化到部署演示的完整实践经验。

通过本次制作，我对智能体系统的设计逻辑、工程实现和实际部署有了更系统的理解。这也是本次作业中最重要的收获之一。
